# ID: 1
- PBI名: プロジェクト管理機能の作成
- ステータス: in progress
- ユーザストーリー
  - ユーザは複数のプロジェクトに所属でき、自分の所属プロジェクト一覧を確認することができる
  - なぜなら、チーム単位でAIKATAを利用できるようにするためだ
- 受け入れ基準
  - ユーザは複数のプロジェクトに所属できる
  - プロジェクト一欄画面で、以下を実施できる
    - 新規プロジェクトの作成
      - 作成する際は、プロジェクト名・説明・APIキー・ユーザを指定できる
        - 説明・APIキーは任意
        - ユーザはデフォルトで自身が選択済み
        - ユーザ指定の際は(大量ユーザが存在することを想定して)検索をすることができる
    - 自分が所属するプロジェクト一覧の確認
      - 各プロジェクトの以下の情報も確認できる
        - プロジェクト名
        - 説明
        - 最終更新日
        - プロジェクトメンバー
          - アイコンで表示、5名以上の場合は超えた分だけ、例えば「+10」などと表示する
    - プロジェクトの検索
- 注意事項
  - 参考にする画面モック:`02-project-list`
  - プロジェクト作成画面は`03b-project-edit`を参考にすること
    - これはプロジェクトの編集画面だが、作成でも参考にはなるはず
  - プロジェクト一覧にて、特定プロジェクトをクリックした後のレビュースペース一欄画面については現時点では対応する必要はない
- 指摘事項
  - サーバ内部でエラーが発生した場合、レスポインスに含まれるエラーを適切に表示できていない箇所がある
    - 全て点検して修正

# ID: 2
- PBI名: 
- ステータス: to do
- ユーザストーリー
  - ユーザがプロジェクト内で、レビュースペースを作成、編集でき、自由に特定のスペースに移動できる
  - なぜなら同一プロジェクト内で、複数のチェックリストを運用できるようにするためだ
    - 例えば、〇〇システム開発プロジェクトがあったとして、開発工程ごとにチェックリストを用意しているはず
      - この場合は開発工程ごとにレビュースペースを作成することができる
- 受け入れ基準
  - 同一プロジェクト内に複数のレビュースペースを作成することができる
  - レビュースペース一覧画面で、以下を実施できる
    - プロジェクト情報の編集
    - レビュースペース情報の確認
      - 最終更新日
  レビュースペース一覧内のサイドバーで、以下を実施できる
    - プロジェクト一覧の確認（サイドバ-上部のプルダウンより）
      - プロジェクトを選択して、そのプロジェクトのレビュースペース一覧画面に遷移もできる
      - （プロジェクトが多いことも想定して）プロジェクトを選択し、絞り込むことができる
      - 大量プロジェクトが存在した場合も、適切にスクロールなどがされて、ユーザ体験が悪化しないようになっている
    - レビュースペース一覧の確認
- 注意事項
  - 参考にする画面モック:`05-review-space-list`
    - モックはサイドバー＋ヘッダの構成となっている。ログイン画面と、プロジェクト関連ページ以外は全てこの構成となるので注意する。
      - layout機能など利用しながら効率的に実装すること
    - パンくずリストについてもログイン画面と、プロジェクト一覧ページ以外は全て共通なので、共通コンポーネントとして定義する
